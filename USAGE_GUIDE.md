## 🎯 如何使用这些文档

### 场景 1: 开发新功能

**步骤**:

1. **阅读功能规格说明书**
   - 找到相关的功能章节
   - 了解现有功能的详细规格
   - 确认新功能与现有功能的关系

2. **查阅数据流程文档**
   - 理解数据如何流动
   - 确认新功能在流程中的位置
   - 评估对上下游模块的影响

3. **填写开发确认检查清单**
   - 找到相关的确认项
   - 在开发前完成所有确认
   - 获得相关人员签字

4. **更新参数配置文档** (如涉及参数)
   - 记录新增参数
   - 更新参数说明
   - 标注修改日期和人员

---

### 场景 2: 修改现有功能

**步骤**:

1. **⚠️ 先确认是否允许修改**
   - 在 `DEVELOPMENT_CHECKLIST.md` 中找到对应模块
   - 检查是否有"不可修改"标记
   - 如有，需要更高层级的审批

2. **阅读相关文档**
   - `FUNCTIONAL_SPECIFICATION.md`: 功能规格
   - `PARAMETER_REFERENCE.md`: 参数说明
   - `DATA_FLOW_DIAGRAM.md`: 数据流程

3. **评估影响范围**
   - 列出所有受影响的模块
   - 确认需要修改的参数
   - 评估对用户的影响

4. **获得确认**
   - 填写检查清单
   - 获得团队成员确认
   - 记录变更原因

5. **实施修改**
   - 按照规格进行修改
   - 更新相关文档
   - 完成测试验证

---

### 场景 3: 代码审查

**审查要点**:

1. **检查是否遵循功能规格**
   - 对照 `FUNCTIONAL_SPECIFICATION.md`
   - 确认实现与规格一致

2. **检查参数使用**
   - 对照 `PARAMETER_REFERENCE.md`
   - 确认参数值在允许范围内
   - 确认没有硬编码的魔法数字

3. **检查数据流程**
   - 对照 `DATA_FLOW_DIAGRAM.md`
   - 确认数据流动正确
   - 确认错误处理完善

4. **检查确认清单**
   - 对照 `DEVELOPMENT_CHECKLIST.md`
   - 确认所有检查项已完成

---

### 场景 4: 问题排查

**步骤**:

1. **定位问题模块**
   - 使用 `DATA_FLOW_DIAGRAM.md` 确定问题发生的位置
   - 查看相关模块的依赖关系

2. **检查相关参数**
   - 使用 `PARAMETER_REFERENCE.md` 查看参数配置
   - 确认参数值是否正确

3. **对比功能规格**
   - 使用 `FUNCTIONAL_SPECIFICATION.md` 确认预期行为
   - 对比实际行为与规格的差异

4. **查看确认记录**
   - 使用 `DEVELOPMENT_CHECKLIST.md` 查看最近的修改
   - 确认是否有相关变更

---

## 📋 关键确认项速查

### 不可修改的核心参数 ⚠️

以下参数修改需要特别审批：

| 参数 | 当前值 | 位置 | 影响 |
|------|--------|------|------|
| GPX 重采样间距 | 20米 | `core/utils.py:17` | 影响所有 GPX 处理 |
| FIT 滤波窗口 | 7点 | `core/utils.py:28` | 影响所有 FIT 处理 |
| 分段长度 | 200米 | `core/predictor.py` | 影响特征粒度 |
| VAM 限制 | 1000 m/h | `core/predictor.py:235` | 影响陡坡预测 |
| 分段表规则 | 起点/CP/终点 | `app.py:582-727` | 影响用户界面 |

### 必须确认的修改流程

1. 任何滤波参数的修改
2. 任何模型参数的修改
3. 任何显示规则的修改
4. 任何数据结构字段的修改
5. 任何验证规则的修改

## 🚨 紧急情况处理

### 发现功能被意外修改

**立即行动**:

1. **停止当前工作**
2. **通知团队成员**
3. **对比文档和代码**
4. **评估影响范围**
5. **决定恢复或确认变更**

**恢复步骤**:

```bash
# 1. 查看最近的提交
git log --oneline -10

# 2. 找到问题提交
git show <commit-hash>

# 3. 如需恢复
git revert <commit-hash>

# 4. 更新文档，记录变更原因
```
